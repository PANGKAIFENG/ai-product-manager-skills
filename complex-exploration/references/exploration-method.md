# Complex Exploration Method

Use this reference when a task is high-stakes, ambiguous, multi-round, or likely to create reusable methodology. The goal is to keep `SKILL.md` lean while preserving the reasoning model behind complex exploration.

## Methodology Anchors

| Anchor | Practical Use In This Skill | Source |
| --- | --- | --- |
| Wicked problems | Treat problem framing as part of the work; do not assume the first question is correct. | Rittel & Webber, `Dilemmas in a General Theory of Planning`: https://www.sympoetic.net/Managing_Complexity/complexity_files/1973%20Rittel%20and%20Webber%20Wicked%20Problems.pdf |
| Double Diamond | Separate discovery / definition from development / delivery; do not jump to final output while the frame is unstable. | Design Council Double Diamond: https://www.designcouncil.org.uk/resources/the-double-diamond/ |
| Cynefin complex domain | In complex domains, probe, sense, then respond; do not impose a simple best-practice answer too early. | Snowden & Boone, HBR: https://hbr.org/2007/11/a-leaders-framework-for-decision-making |
| Premortem / prospective hindsight | Before committing to a plan, imagine it failed and identify likely causes. | Gary Klein, HBR: https://hbr.org/2007/09/performing-a-project-premortem |
| After Action Review | After the work, compare intended outcome, actual outcome, causes, and next-time changes. | U.S. Army AAR appendix: https://www.first.army.mil/Portals/102/FM%207-0%20Appendix%20K.pdf |

Do not cite these frameworks as decoration. Use them to choose behavior: slow down framing, expose assumptions, run probes, pre-mortem plans, and convert the process into reusable assets.

## Common Failure Modes

| Failure Mode | Symptom | Defense |
| --- | --- | --- |
| Premature execution | User asks for Roadmap, PRD, price table, or PPT and the agent immediately writes it. | Start with task type, hidden assumptions, and real problem. |
| Narrow question trap | The prompt asks about one feature, but the real issue is packaging, strategy, positioning, or system shape. | Reframe from object to managed system. |
| Evidence pile | Research output lists many facts but no decision implication. | Tie each evidence type to a question it answers. |
| Framework sprawl | The agent adds frameworks without deciding which one governs the work. | Offer 2-3 frameworks, recommend one, state failure conditions. |
| Version churn | Many revisions happen, but nobody distinguishes deeper insight from surface rewrite. | Classify each iteration as problem upgrade, framework rebuild, judgment correction, expression polish, detail polish, boundaryless expansion, or compliance-only execution. |
| Asset loss | Final artifact exists, but no reusable method, template, or prompt remains. | End major work with five asset types. |

## Task-Type Decision Notes

- Use `execution` only when the output target, scope, and format are already clear.
- Use `solution design` when the main uncertainty is which path to take.
- Use `positioning` when the object, category, differentiated promise, or mental model is unstable.
- Use `system` when the work spans roles, modules, processes, data, runtime, or layered capabilities.
- Use `strategy` when it affects sequencing, investment, pricing, packaging, competitive advantage, or organizational priority.
- Use `methodology` when the user wants reusable patterns from a process.
- Use `cognitive calibration` when the highest-value move is challenging whether the question should be asked this way.

## Probe Design

When the domain is complex, plan probes instead of pretending to know the answer:

1. Name the uncertain assumption.
2. Define the smallest evidence that would reduce uncertainty.
3. Decide what output will change if the assumption is true or false.
4. Set a stop condition for the probe.
5. Preserve reusable learning even if the probe disproves the initial direction.

Probe examples:

- Roadmap: compare two capability-layer maps before writing a monthly plan.
- Pricing: test whether the charging object is feature access, asset usage, governance, AI completion, or implementation service.
- Competitive positioning: compare onboarding, switching cost, data moat, and workflow depth instead of copying feature lists.

## Asset Extraction Quality Bar

A useful asset must be reusable without the full original chat. It should include:

- Trigger: when to use it again.
- Inputs: what material it needs.
- Steps: what to do in order.
- Decision rule: how to judge or choose.
- Output shape: what artifact it produces.
- Failure signs: when not to use it.

If an asset lacks trigger, input, decision rule, or output shape, keep it as a note rather than calling it a method.
