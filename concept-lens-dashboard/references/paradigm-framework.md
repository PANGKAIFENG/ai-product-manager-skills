# Paradigm Framework

## Core Principle

Do not use a timeline as the final structure. First recover the concept's lineage, then build stages from the conflict between rising scale/complexity and the user's desired control granularity. A real paradigm shift changes what the system treats as the primary object of control.

Historical context is a calibration tool, not the end product. Use it to prevent present-day vendor categories, fashionable terminology, or generic market narratives from overwriting the concept's original problem and later semantic shifts.

Examples:

- Search: from keyword match to semantic intent matching.
- CRM: from static customer records to workflow ownership to predictive orchestration.
- Warehouse operations: from location bookkeeping to flow optimization to adaptive fulfillment networks.

## Stage Induction

Use this sequence:

1. Identify the user's real job and the field's scarce resource.
2. Trace the concept lineage: earliest available context, original problem, original users, original use case, and original contradiction.
3. List historical or industry solution families with source evidence.
4. Map semantic shifts: technical adoption, business packaging, cross-domain reuse, standardization, and common misuse.
5. Group solution families by the control object they optimize.
6. Remove decorative stages that do not change the control object.
7. Keep 3-6 stages only when each stage has a distinct contradiction and solution pattern.

Reject "four stages by default." If only two real paradigms exist, say so. If six are justified, use six.

## Concept Lineage Lens

Before drafting the matrix, answer these questions:

- Origin context: where and why did this concept emerge, and how confident is the evidence?
- Original use: who used it first, for what decision or task, and what was the promised improvement?
- Original contradiction: what mismatch, bottleneck, or control problem made the concept necessary?
- Evolution points: which standards, products, operating practices, or business shifts changed the meaning?
- Semantic drift: what does the term mean today that it did not originally mean?
- Misuse or overextension: where is the concept used as a label for something that does not change the control model?

If the origin is contested or poorly sourced, state the uncertainty and use the earliest reliable context rather than inventing a clean origin story.

## PM Matrix Dimensions

For each stage, fill these dimensions:

- 典型场景与诉求: the concrete business situation and the user's rigid or elastic demand.
- 解决的本质矛盾: the conflict between real-world complexity and the system's capacity or control model.
- 行业标准解决方案: mature architecture, workflow, product pattern, or business model used in this stage.
- PM 必备的高层逻辑: explain the operating mechanism through first principles, probability, economics, geometry, systems theory, or physics.
- 源流证据与语义变化: what historical evidence or semantic shift supports this stage boundary.
- 平等对话黑话/切入点: 1-2 precise review questions that expose assumptions, hidden debt, or architecture mismatch.

## Debt Detector

Find 2-3 cases where a low-stage tool is forced to support high-stage demand. For each case:

- Symptom: what users or operators experience.
- Root mismatch: which demand belongs to a later paradigm.
- False fix: the common patch that increases debt.
- Breakthrough: the cleaner architectural, workflow, or operating-model shift.
- PM review question: a short question that can be used in a review.

## Cognitive Density Rules

- Open with a direct essence statement, not a generic preface.
- Use one strong metaphor when it clarifies the mechanism.
- Prefer mathematical, physical, economic, or geometric metaphors over vague analogies.
- Avoid inflated claims like "top 1%" in the output; demonstrate depth through specificity.

## Banned Patterns

Avoid:

- "随着时代的发展"
- "技术是一把双刃剑"
- "降本增效" without explaining the actual mechanism
- parameter lists with no decision meaning
- stage names that only rename the same operating model
- stage boundaries based only on publication dates or vendor release timelines
- present-day definitions projected backward as if they were the original meaning
- origin stories with no source or uncertainty label
- review questions that could apply to any architecture
