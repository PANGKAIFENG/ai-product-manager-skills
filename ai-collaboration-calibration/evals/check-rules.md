# 评测检查规则

## 评测分层

| 层 | 方法 | 检查内容 |
|----|------|---------|
| 确定性检查 | 关键词匹配 | 响应中是否包含必要结构 |
| Rubric 评分 | 模型辅助评分 | 假设识别质量、领域定位准确性、是否过度校准 |
| 回归验证 | 每次修改后跑全集 | 15 条是否仍通过 |

## 确定性检查规则

### 触发场景（T01-T06）必须包含

- [ ] "问题分类" 或 "问题类型"（含 6 种类型之一的关键词）
- [ ] "假设" 或 "隐含"
- [ ] "领域" 或 "术语"
- [ ] "层级" 或 "L1/L2/L3/L4"
- [ ] 不包含具体实现方案（首次响应）

### 非触发场景（T07-T10）必须不包含

- [ ] 不包含 "校准结果"
- [ ] 不包含 "隐含假设"
- [ ] 不包含 "问题分类"
- [ ] 不包含 "协作层级"
- [ ] 直接给出执行结果

## Rubric 评分模板

对每条触发测试，用以下 rubric 评分（1-5 分）：

### 假设识别质量（1-5）

| 分数 | 标准 |
|------|------|
| 1 | 没有识别假设 |
| 2 | 识别了假设但都是表面的（用户已明说的） |
| 3 | 识别了 1 个真正隐含的假设 |
| 4 | 识别了 2+ 个隐含假设，每个有验证方式 |
| 5 | 识别了关键假设，且指出了假设之间的依赖关系 |

### 领域定位准确性（1-5）

| 分数 | 标准 |
|------|------|
| 1 | 没有做领域定位 |
| 2 | 给了泛化分类（"产品问题"、"技术问题"） |
| 3 | 给出了领域名但没有标准解 |
| 4 | 给出了可检索的领域术语 + 标准解 |
| 5 | 给出了术语 + 标准解 + 当前方案与标准解的差距 |

### 过度校准检查（pass/fail）

| 检查项 | 通过标准 |
|--------|---------|
| L1 任务未被拉到 L4 | T07-T10 直接执行 |
| 校准深度和问题复杂度匹配 | 简单问题不做全流程校准 |
| 用户要求停止时及时停止 | 不强制继续校准 |

## 运行方式

### 手动测试流程

1. 安装 Skill 到 Claude Code
2. 逐条输入 T01-T06，检查响应是否满足确定性检查
3. 逐条输入 T07-T10，确认不误触发
4. 对 T01-T06 的响应做 Rubric 评分
5. 记录每条的通过/失败状态

### 修改后回归

每次修改 SKILL.md 后，重跑全部 15 条，记录变化。
