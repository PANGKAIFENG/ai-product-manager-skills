# 06 反向推演（失败预演）

## 适用场景

已有问题框架，但仍担心问题定义、用户需求、约束或判断标准会导致后续方案失败。这个模式用于校准阶段的风险感知，不用于成熟方案的连续压力测试。

## 不适用场景

- 方案还在萌芽阶段、细节不足以做有意义推演
- 问题定义还不清楚（应先做 05 问题分类 + 11 提升问题层级）
- 问题已确认，用户已有具体方案 / 架构 / 计划 / 决策，并希望一问一答压测取舍和失败模式（应转用 `grill-me`）

## Prompt 模板

```
假设 3 个月后这个方向失败了，请做一次问题定义层面的失败预演：
列出最可能的 5-8 个失败原因，并按以下维度分类：
问题定义错误 / 用户需求误判 / 技术架构错配 / 数据质量不足 / 工程资源不足 / 评测指标缺失
最后指出最应该提前验证的 3 个风险。
```

## AI 输出要求

1. 列出 5-8 个失败原因
2. 按 6 个维度分类
3. 指出最应提前验证的 3 个风险
4. 每个风险给出验证方式

## 验收标准

- 失败原因是具体的（不是"可能做不好"）
- 分类覆盖多个维度（不只是技术维度）
- 最应验证的 3 个风险有明确的验证方式

## 常见误用

把推演出的失败场景当作已发生的事实，导致过度防御、不敢行动。失败预演是为了提前验证，不是为了否定方案。
