# 05 问题分类

## 适用场景

在开始任何方案之前，判断这属于哪一类问题。

## 不适用场景

- 问题已经被充分定义和分类
- 纯执行类任务（翻译、格式化等）

## Prompt 模板

```
在我开始设计方案之前，请先帮我判断这属于哪一类问题：
已知解问题 / 设计问题 / 发现问题 / 棘手问题 / 组织问题 / 数据治理问题
用一句话说明判断依据，以及这个分类对方案设计的最大含义是什么。
```

## 问题类型速查

| 问题类型 | 特征 | 正确方法 |
|---------|------|---------|
| 已知解问题 | 别人解决过了 | 找领域 → 找最佳实践 → 适配约束 |
| 设计问题 | 知道要什么，不知道怎么实现 | 迭代原型，在行动中澄清 |
| 发现问题 | 连要什么都不确定 | 快速探索，用小实验压缩不确定性 |
| 棘手问题 | 解决会改变问题定义 | 边解边定义，接受漂移 |
| 组织问题 | 阻力来自人和流程而非技术 | 对齐利益相关方，处理动机问题 |
| 数据治理问题 | 技术方案正确但数据质量决定效果 | 先解决数据问题，再优化模型 |

## AI 输出要求

1. 明确判断属于哪一类（可以是混合类型，但要说明主次）
2. 一句话判断依据
3. 这个分类对方案设计的最大含义
4. 如果是混合类型，说明哪个方面用哪种方法

## 验收标准

- 分类有明确依据，不是凭感觉
- 分完类后给出了对应方法（不是所有类型都用"分析→设计→执行"）
- 分类对后续行动有实际影响

## 常见误用

分完类就直接进入方案设计，跳过领域定位和裂缝定位。分类是起点，不是终点。
