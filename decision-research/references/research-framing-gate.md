# Research Framing Gate (R00)

## 目的

防止"过早下钻"——用户提到一个具体词（桌面端、Computer Use、本地资产治理），AI 立刻围绕它展开搜索，而没有先判断它在整个研究中的层级位置。

## 三步判断

### Step 1: 用户给的是什么

| 类型 | 识别信号 | 处理 |
|------|---------|------|
| 问题 | 「什么方案能…」「选哪个…」「怎么做…」 | 直接进 R01 |
| 现象 | 「不好用」「好像该做」「有点慢」「体验不行」 | 追问：背后要解决什么决策？ |
| 解法 | 「接入 X」「做 Y 功能」「用 Z 框架」 | 上提一层：它解决什么更大的问题？如果不做它，问题还存在吗？ |

### Step 2: 研究层级

| 层级 | 代表问题 | 典型研究范围 |
|------|---------|-------------|
| L1 行业叙事 | 主流 Agent 产品在宣称什么价值？ | 10+ 产品对外话语、行业报告、投资方向 |
| L2 产品定位 | 我们的产品在行业中扮演什么角色？ | 竞品映射、用户群体、差异化假设 |
| L3 业务场景 | 哪些工作流能体现差异？ | JTBD、用户任务、交付物、流程 |
| L4 能力体系 | 需要什么底层能力？ | 模块、接口、权限、数据、工具 |
| L5 技术实现 | 选哪个方案？怎么做？ | 框架、库、API、架构、代码 |
| L6 商业包装 | 怎么卖？怎么升级？ | 版本分层、定价、套餐、渠道 |

**关键规则**：如果用户给的是 L5 的词（如"Computer Use""本地文件索引"），但问题层级实际在 L2（桌面端存在的意义），必须先回到 L2 框定后再下钻。

### Step 3: 研究类型

| 类型 | 识别词 | 默认模式 |
|------|--------|---------|
| 技术选型 | 选哪个、用什么、A vs B | 标准决策流 |
| 可行性验证 | 能不能、可不可以、有没有限制 | Spike + 最小证据 |
| 平台/API 接入 | 怎么接、有没有 API、SDK | 官方优先 + 合规拦截 |
| 产品策略 | 方向对不对、怎么定位、为什么做 | Top-Down Mode |
| 竞品判断 | 别人怎么做、行业怎么看 | Top-Down Mode |
| 商业模型 | 怎么卖、怎么分层、升级价值 | Top-Down Mode |
| 行业格局 | 主流叙事、趋势、价值主张 | Top-Down Mode |

## 输出格式

```
**Framing**
- 用户给的是：[问题 / 现象 / 解法]
- 研究类型：[技术选型 / 可行性 / 平台接入 / 产品策略 / 竞品判断 / 商业模型 / 行业格局]
- 当前层级：[L1-L6]
- 是否需要上提：[是 → 上提到 LX / 否]
- 启用模式：[标准决策流 / Top-Down Product Research Mode]
```

## 失败模式（必须避免）

1. 用户说"做桌面端"→ 直接搜索桌面端功能列表（应先问桌面端为什么存在）
2. 用户说"本地款式资产治理"→ 直接搜索 DAM 方案（应先问它服务什么更大方向）
3. 用户说"接 Computer Use"→ 直接搜索 Computer Use 实现（应先区分执行能力层级）
4. 用户给了多个不同层级的词 → 平等对待全部展开（应先标注层级，建立层级树）

## 何时跳过 R00

- 用户的问题已经是一个明确、可验证的技术问题（如"Python 的 asyncio 和 threading 选哪个"）
- 用户明确说"不需要讨论方向，直接帮我选方案"
- 问题层级在 L5 且没有上层歧义
