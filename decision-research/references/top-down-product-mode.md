# Top-Down Product Research Mode

## 触发条件

R00 判断满足以下任一条件时自动启用：

- 研究类型为：产品策略 / 竞品判断 / 商业模型 / 行业格局
- 研究层级 ≥ 产品定位（L2）
- 用户给的是解法或现象，需要先上提建立上层框架

## 核心规则

**不允许在上层结论未建立时直接跳到下层。**

产品研究从高到低逐层下钻，每一层的结论支撑下一层的研究范围：

```
L1 行业叙事 → L2 产品定位 → L3 业务场景 → L4 能力体系 → L5 技术实现 → L6 商业包装
```

## 每层的核心问题

### L1 行业叙事

- 这个品类里主流产品在对外宣称什么价值？
- 行业叙事正在从什么转向什么？
- 哪些叙事有实际产品支撑，哪些只是概念？

### L2 产品定位

- 我们的产品在行业中扮演什么角色？
- 我们服务谁？和相邻产品的差异在哪里？
- 用户为什么选我们而不选替代品？

### L3 业务场景

- 目标用户的哪些工作流能体现我们的差异？
- 每个工作流的输入、判断、输出和交付物是什么？
- 哪些场景只有我们能做好，哪些是红海？

### L4 能力体系

- 支撑这些场景需要什么底层能力？
- 能力之间的依赖关系是什么？
- 哪些能力是通用的，哪些是垂直的？

### L5 技术实现

- 每个能力的实现选项是什么？
- 风险、复杂度、时间线？
- 先做什么、后做什么？

### L6 商业包装

- 这些能力怎么打包成产品版本？
- 不同版本之间的升级价值是什么？
- 用户为什么愿意付更多？

## Deliverable-Backwards Pattern

产品研究中强制从交付物倒推能力：

```
用户类型 → 角色 → JTBD → 输入资产 → 最终交付物 → 工作流步骤 → 所需能力 → 人审点 → Surface fit
```

这防止了"从已有功能往外包装"的错误方向。

## Anti-Overclaim Checklist

产品策略研究必须检查：

- [ ] 区分 current 能力 vs roadmap 承诺
- [ ] 区分 SaaS 能做 vs Desktop 必须做
- [ ] 区分 基础版 vs 高级版
- [ ] 区分 本地保存 vs 完全离线
- [ ] 区分 授权浏览器研究 vs 自动抓取
- [ ] 区分 文件处理 vs 任意 Computer Use
- [ ] 区分 自动化建议 vs 自动发布
- [ ] 明确人审确认和审计边界

## 输出格式

每完成一层后输出：

```markdown
**L[N] 结论**
- 核心发现：[2-3 句]
- 对下一层的影响：[下一层的研究范围因此变为…]
- 关键证据：[来源 + 层级]
- 排除项：[这一层排除了什么假设]
```

## 失败模式

- 用户说"桌面端"→ 直接进入 L5 搜索桌面端技术方案（应从 L1-L2 开始：桌面端在行业里扮演什么角色）
- 用户说"推款闭环"→ 直接搜索推款流程实现（应先问推款是不是真正的差异点）
- 用户给了 5 个不同层级的点 → 并行展开（应先标注层级，建立问题树，从最高层开始）
