# 有立场结论模板

## 核心结构

```
推荐：[X]（置信度：高 / 中 / 低）
  核心依据：[2-3 句]，来源 [L1/L2]

排除：
  - [Y]：[具体排除理由 + 证据]
  - [Z]：[具体排除理由 + 证据]

前提：以上结论建立在 [假设 A] 成立的基础上

颠覆条件：如果出现 [具体信号 B]，这个结论需要重新评估

待验证：[单一来源或只有 PoC 才能确认的结论]
```

## 置信度标签定义

| 置信度 | 条件 |
|--------|------|
| **高** | ≥2 个独立 L1/L2 来源支持，无重大反对证据 |
| **中** | 有 L1/L2 来源，但存在未解决的竞争假设 |
| **低** | 主要来自 L3/L4，或核心假设未验证 |

## Evidence vs Interpretation Split

每条关键判断标注来源性质：

| 标签 | 含义 | 可信度 |
|------|------|--------|
| `[Evidence]` | 外部公开资料直接支持 | 可追溯验证 |
| `[Local Context]` | 本地项目事实 | 已确认 |
| `[Inference]` | 基于证据的推断 | 逻辑链可审查 |
| `[Judgment]` | 当前产品/战略判断 | 可能随新信息改变 |
| `[Open]` | 仍未决定 | 需后续验证 |

示例：

```
- 桌面端差异不在推款结果，而在执行过程 [Judgment]
- OpenAI Operator 用浏览器完成重复性网页任务 [Evidence, L1]
- StyleClaw Desktop 应定位为受控执行工作台 [Inference, 基于行业叙事迁移]
- 用户是否接受"本地工作区+在线计费"的模式 [Open, 需用户验证]
```

## 颠覆条件要求

颠覆条件是必填项。好的颠覆条件：具体、可观察、和结论有因果关系。

- 好：「如果 iLink 公测结束后不对外开放，需要切换到企业微信方案」
- 好：「如果团队 3 个月内扩到 10 人以上，方案 B 的运维成本会超过方案 A」
- 差：「如果情况发生变化，结论可能需要调整」（无意义）

## 矩阵的使用边界

合理场景：用户明确说「帮我比较多个方向，决策权在我」。

即便如此，矩阵后必须附立场：
```
如果是我：在你描述的约束下，我选 B，原因是…如果你的约束不同，告诉我哪里不一样。
```

## 暗知识缺口下的结论

当终止条件 R10 触发时：

```
当前可以确认：[基于公开信息的判断]（置信度：中）

无法通过二手调研确认：[情境变量 X]

建议下一步：
- 做什么：[最小验证动作]
- 验证什么：[要回答的具体问题]
- 预计成本：[时间/资源估算]
```

## Assumption Ledger 附录

如果调研过程中有用户纠偏，结论末尾附上 Assumption Ledger 摘要：

```
**调研过程纠偏记录**
| # | 旧假设 | 用户纠正 | 新判断 |
|---|--------|---------|--------|
```
