# 渠道选择手册

## 核心原则

不是所有渠道全开。先判断问题类型，再选渠道。选定渠道后，每条信息必须标注来源层级。

## 来源层级体系

| 层级 | 定义 | 示例 |
|------|------|------|
| **L1** | 官方第一手来源 | 官方文档、官方 SDK、原始 API 规范、官方 Changelog、官方 RFC、SEC 文件 |
| **L2** | 独立可验证来源 | 独立第三方技术报告、学术论文、权威机构研究、行业分析师报告 |
| **L3** | 社区主流实现 | 高依赖数量的开源项目、活跃维护的社区实现、结构化市场信号 |
| **L4** | 二手经验来源 | 工程师博客、技术文章、论坛讨论、X/Twitter 观点、媒体报道 |

混用规则：
- 核心结论必须有 L1 或 L2 来源支撑
- L3/L4 只能作为补充和佐证
- 单独的 L4 来源不能构成结论，只能标为「待验证假设」

## 按问题类型选渠道

### 平台 / API 接入类

首选渠道：官方文档 → 官方 SDK → 官方 Changelog → 官方 issues/discussions

特殊规则 — 合规前置拦截：逆向协议直接排除，不参与比较。

### 技术选型类

首选渠道：官方文档、GitHub（dependent repos + issue 活跃度）、独立技术报告

GitHub 质量评估指标：
- `dependent repositories`：有多少项目依赖它
- `issue 响应速度`：最近 issues 有没有维护者回复
- `最近 commit 频率`：过去 3 个月有没有活跃维护
- `bus factor`：核心贡献者是否过度集中

禁止：用 stars 数量作为选型的主要依据。

### 可行性验证类

首选渠道：官方文档、官方示例/Cookbook、最小可运行示例

Spike 思路：找最小可验证路径，不是完整解决方案。

### 竞品 / 参考实现类

首选渠道：公开代码库、官方博客、产品文档、changelog

关键规则：先向用户确认参考对象名，再搜索。

### 产品策略 / 产品定位类

首选渠道：
1. 官方产品页 / About / 使命宣言（L1）
2. 官方博客 / CEO/创始人文章（L1/L2）
3. 产品 changelog / release notes（L1）
4. 行业分析报告 / 研究机构（L2）
5. 投资者材料 / 融资新闻（L2/L3）
6. 招聘信息（L3/L4 弱信号）

禁止：把个人博客对产品的评价当成产品定位的证据。

### 商业模型 / 版本分层类

首选渠道：
1. 定价页（L1）
2. 功能对比表 / plan comparison（L1）
3. 官方案例/ROI 报告（L1/L2）
4. SEC 文件 / 年报（L1）
5. 行业分析师报告（L2）
6. 招聘信息 / 团队结构（L3/L4 弱信号）

### 行业格局 / 品类研究类

首选渠道：
1. 各竞品官方产品页（L1）
2. Product Hunt / G2 / Capterra（L3）
3. 行业报告 / Gartner / Forrester（L2，可能付费）
4. 行业大会演讲 / 官方案例（L2/L3）
5. 媒体深度报道（L3/L4）
6. X / HN / Reddit 讨论（L4 弱信号）

### 学术方法 / 算法类

首选渠道：arXiv、Semantic Scholar、Papers with Code、官方技术报告

注意：AI/ML 领域 3 年以上的论文结论可能已被推翻。

### 趋势 / 社区观点类

首选渠道：HN、Reddit、X、技术 Newsletter

强制规则：所有趋势类来源默认标注为弱证据，必须交叉验证。

## 产品/市场研究补充渠道

| 渠道 | 适合 | 证据等级 | 注意 |
|------|------|---------|------|
| Product Hunt | 新产品发现、launch 定位 | L3/L4 | 关注 maker comments |
| G2 / Capterra | B2B 产品评论、buyer pain | L3 | 有 review bias |
| AlternativeTo | 替代品发现 | L4 | 分类松散 |
| App Store / Google Play | 移动端评论 | L3 | sampling bias |
| Chrome Web Store / VS Code Marketplace | 工具类产品 | L3 | 评论可能少 |
| 定价页 | 商业模型、打包方式 | L1 | 价格会变，标注日期 |
| 产品 Changelog | 产品进化方向 | L1 | 信息选择性披露 |
| Demo 视频 / Webinar | 产品工作流 | L4 | 推广性质 |
| 用户论坛 / 社区 | 痛点和工作流 | L3/L4 | 轶事性 |
| 招聘信息 | 路线图暗示 | L4 | 间接证据 |
| BuiltWith / SimilarWeb | 技术栈和流量估算 | L4 | 估算可能不准 |

## 来源质量快速检查

每条信息进入假设矩阵之前：

1. 发布者是谁？
2. 发布时间？（>1 年需验证时效性）
3. 利益关系？（厂商自比有偏差）
4. 可验证性？（能否独立验证）

通不过则降级到 L4，标注为「待验证」。
