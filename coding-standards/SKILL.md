# 编码规范与工程最佳实践（coding-standards）

## 中文速查

- 中文名：编码规范 / 工程规范
- 英文稳定名：`coding-standards`
- 你可以这样叫我：`按编码规范实现`、`帮我检查代码规范`、`这个实现有没有工程坏味道`、`别过度设计`
- 适合：TypeScript、JavaScript、React、Node.js 的实现约束、代码评审、重构建议和质量基线
- 不适合：项目专属架构、目录、测试命令或业务约束；这些应放到项目级 Skill

## Overview

适用于多数项目的通用编码规范。目标是让实现保持清晰、类型安全、可测试、不过度设计，并且能落到具体代码行为，而不是输出泛泛原则。

## Output And Deliverables

使用这个 Skill 时，最终交付物应是可直接落到代码或评审意见里的工程约束：

- 写代码：实现结果体现类型安全、清晰命名、明确错误处理、可测试边界和最小必要抽象。
- 改代码：说明改动如何降低复杂度、重复、隐式状态或不可维护风险。
- 做 review：输出具体文件/位置、问题影响、建议修复方式和必要测试。
- 定规范：给出团队可执行的规则、反例、例外条件和迁移建议。

## Core Principles

- 可读性优先：变量、函数、模块边界要表达意图。
- KISS：用能工作的最简单方案，避免炫技和过早优化。
- DRY：抽出真实重复，不为猜测中的未来重复提前抽象。
- YAGNI：没有真实需求时，不引入复杂框架、目录迁移或通用层。
- 类型安全：避免 `any` 扩散，输入输出边界要有明确类型或 schema。
- 错误处理：关键失败路径要显式处理，并保留可调试信息。
- 可测试：关键业务路径要有测试，或至少给出可执行的测试建议。

## Workflow

1. 先确认任务类型：写代码、改代码、做 review，还是制定团队规范。
2. 读取项目已有规范、lint、typecheck、test 命令；如果没有项目约束，再使用本 Skill 的通用基线。
3. 按任务需要加载 `references/engineering-rules.md` 的对应章节。
4. 输出能直接落到代码、评审意见或团队规则里的建议，并说明必要验证方式。

## Resource Guide

按任务需要加载 `references/engineering-rules.md`：

- 写 TypeScript / JavaScript：看命名、不可变更新、async/await、类型安全、错误处理。
- 写 React：看组件 props、hooks、状态更新、条件渲染。
- 设计 API：看 REST 约定、统一响应、输入校验。
- 做代码组织：看文件命名、目录职责、注释和文档规则。
- 做性能或测试 review：看 memoization、lazy loading、数据库查询、AAA 测试结构。
- 找工程坏味道：看长函数、深层嵌套、魔法数字等反模式。

## Acceptance Criteria

- 代码能通过项目已有 lint、typecheck、test 或等价检查。
- 没有为了“规范”引入与需求无关的抽象、框架或目录迁移。
- 关键失败路径有错误处理，关键业务路径有测试或明确测试建议。
- 评审意见能落到具体代码行为，不停留在“可读性更好”“建议优化”这类空泛表达。

## Evaluation

Smoke prompts:

- `按编码规范实现这个 TypeScript 函数。`
- `帮我 review 这段 React 代码有没有工程坏味道。`
- `这个 API response 设计是否稳定？`

Non-trigger prompts:

- `解释这个产品需求。`
- `给这个项目写一份专属架构规范。`
- `帮我画架构图。`
