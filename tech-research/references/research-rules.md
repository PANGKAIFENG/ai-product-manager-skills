# 调研规则集（R01-R11）

## 启动协议

### R01 — 决策问题锚定

**核心**：调研服务于决策，不是服务于信息完整性。

操作：AI 把用户描述里隐含的决策问题说出来，同时确认终止条件。

**识别信号（需要触发 R01）**：
- 用户描述的是一个研究主题而不是一个决策（「我想了解 X」而不是「我需要决定是否用 X」）
- 用户没有说调研结束后要做什么

**格式**：
> 「你问的是 [X]，我理解你的决策是 [具体判断/行动]。这个理解对吗？调研完你需要知道什么才能往下走？」

**失败信号**：跳过 R01 → 调研结束后用户说「但这不是我想知道的」。

---

### R02 — 假设显式化

**核心**：把 AI 当前持有的倾向性假设列出来，预先定义「反转信号」。

操作：在开始搜索前，列出预设并问「如果 [假设 A] 是错的，我应该看到什么信号？」

**为什么关键**：没有预先定义反转信号，调研必然滑向确认偏差——找到的全是支持当前假设的证据。

**格式**：
> 「在开始之前，我注意到这几个预设：
> 1. [假设 A]
> 2. [假设 B]
> 哪个你不确定？如果 [假设 A] 是错的，我们应该看到什么样的证据？」

**失败信号**：调研结果和用户直觉完全一致，没有任何新发现 → 可能是确认偏差，不是调研质量高。

---

### R03 — 暗知识缺口识别（Ground Truth 核对）

**核心**：AI 天然只能获取公开的「廉价地表水」，必须主动获取用户的隐性知识。

操作：搜索前必问两个问题：
1. 「你有没有见过把这个做得好的案例，或者已有的参考实现？」
2. 「有没有你已经排除的方向？」

**为什么关键**：用户可能已经有了最好的答案（比如直接指向了一个参考项目），跳过这步等于在信息非对称的基础上做无效调研。

**操作规则**：
- 拿到参考实现 → 优先深挖，不开启 Web 搜索
- 拿到排除方向 → 记录为「已排除假设」，不浪费搜索资源
- 用户说「不知道」→ 正常进入搜索流程

**失败信号**：调研结束后用户说「其实我们项目里已经有类似实现了」或「这个方向我们之前就否定过了」。

## 执行规则

### R04 — 竞争假设枚举（禁止单假设调研）

**核心**：第一个搜到的结果会无意识成为基准（锚定效应），提前枚举竞争假设可以打破这个偏差。

操作：任何搜索之前，先输出：
> 「当前有 N 个可能的结论候选，分别是：[H1]、[H2]、[H3]。」

**最低要求**：至少 2 个互斥假设。「这个技术方向可行」和「这个技术方向不可行」不算互斥假设，那只是一个假设的正反面。

**失败信号**：整个调研过程只分析了一个方向，其他方向没有被显式排除，只是被忽略。

---

### R05 — 渠道动态选择

**核心**：不是所有渠道全开，先判断问题类型，再动态选渠道。

**渠道优先级规则**：

| 问题类型 | 首选渠道 | 禁止动作 |
|---------|---------|---------|
| 平台/API 接入 | 官方文档 → 官方 SDK → 官方 Changelog | 逆向协议直接排除，不参与比较 |
| 技术选型 | 官方文档、GitHub（dependent repos）| 不用 stars 作为主要质量指标 |
| 可行性验证 | 官方文档、最小可运行示例 | 不用博客帖子验证技术边界 |
| 竞品/参考实现 | 公开代码库、官方博客 | 先确认参考对象名，防止找错项目 |
| 趋势/观点 | HN、Reddit、X | 必须标注为弱证据，单独不构成结论 |

**来源层级标注（每条信息必须标注）**：
- L1：官方文档 / 官方 SDK / 原始文件
- L2：独立第三方研究 / 受信任技术报告
- L3：社区主流实现（依赖数量 + issue 响应速度）
- L4：从业者经验 / 博客 / 二手摘要

**失败信号**：结论引用的全是 L4 来源，没有任何 L1/L2 交叉验证。

---

### R06 — 反对证据搜索（Devil's Advocacy）

**核心**：主动搜索反对证据，而非支持证据。这是最反直觉也最重要的规则。

操作：对每个假设构造反向问题：
> 「如果 [假设 H] 是错的，我应该看到什么样的证据？」

然后去搜索那个证据。

**诊断性证据标准**：
- 有价值的证据：在假设 A 成立时「很可能出现」，在假设 B 成立时「几乎不会出现」
- 无贡献的证据：在所有假设下都「合理存在」（不能区分假设）

**失败信号**：
- 找不到任何反例 → 搜索空间太窄，或者结论过早收敛
- 所有证据都支持同一个结论 → 很可能只搜索了支持性关键词

---

### R07 — 里程碑推演（禁止跳阶段）

**核心**：不在 V1 阶段跳到 V4 结论。每个里程碑有明确的输入和输出。

**四个里程碑**：

| 里程碑 | 输入 | 核心问题 | 输出 |
|--------|------|---------|------|
| V1 问题成形 | 用户描述 | 我们实际在问什么 | 可验证的研究问题 |
| V2 假设矩阵 | 竞争假设 | 哪些已有证据，哪些尚无 | 假设 × 证据状态表 |
| V3 排除轮 | 诊断性证据 | 哪些假设被排除了 | 排除列表 + 理由 |
| V4 残差分析 | 剩余假设 | 结论的置信边界在哪里 | 结论 + 不确定性说明 |

**失败信号**：第一轮搜索就直接输出推荐方案，没有经过假设枚举和排除过程。

---

## 终止规则

### R08 — 三角收敛终止

**条件**：剩余假设 ≤ 1 个，且有 ≥ 2 个独立来源（不同渠道、不同利益方）支持。

**「独立」的定义**：同一公司的两篇博客不算独立；官方文档 + 独立工程师的实践报告才算独立。

---

### R09 — 信息饱和终止

**条件**：连续 3 条新证据在假设矩阵中没有新的排除能力（都是「与已有结论一致」或「不相关」）。

**操作**：每找到一条新证据，先问「它排除了哪个假设？」如果答案是「没有」，计入饱和计数。

---

### R10 — 暗知识缺口终止

**条件**：不确定性的核心是「情境变量」（团队规模、历史技术债、决策者风格、组织约束）。

**操作**：宣布调研到达边界，转向用户追问或建议 PoC：
> 「到这里我的调研到了边界——这个结论依赖 [情境变量 X]，只有你实际验证才能确认。建议做一个最小 PoC：[具体建议]。」

**失败信号**：继续扩大搜索范围，但搜到的全是通用建议，和用户具体情境无关。

**ROI 熔断触发点**：调研进行到约 50% 时，如果已有证据的置信度不足以支持决策，触发范围评估——缩小决策问题或定向补充，而非全面扩展。

---

## 输出规则

### R11 — 有立场结论（禁止无立场方案矩阵）

**核心**：「给矩阵让用户自选」是调研不够深的表现，不是客观中立。

**必要结构**（来自 SAT Key Judgments 规范）：

```
推荐：[X]（置信度：高/中/低）
  核心依据：[原因]，来源 [L1/L2]

排除：[Y]，因为 [具体原因]
     [Z]，因为 [具体原因]

前提：结论建立在 [假设 A] 成立的基础上

颠覆条件：如果出现 [信号 B]，这个结论需要重新评估

待验证：[单一来源或需要一手验证的结论]
```

**置信度标签**：
- 高：≥2 个独立 L1/L2 来源，无重大反对证据
- 中：有 L1/L2 来源，但存在未解决的竞争假设
- 低：主要来自 L3/L4，或核心假设未验证

**「颠覆条件」是必填项**：没有颠覆条件的结论等于把适用边界隐藏了，形式上有立场，实际上还是把判断推给用户。

**矩阵的唯一合理使用场景**：用户明确说「帮我比较多个方向，决策权在我」。即便如此，矩阵后必须附「如果是我，我选 X，因为……」的立场。
