# 渠道选择手册

## 核心原则

不是所有渠道全开。先判断问题类型，再选渠道。选定渠道后，每条信息必须标注来源层级。

---

## 来源层级体系

| 层级 | 定义 | 示例 |
|------|------|------|
| **L1** | 官方第一手来源 | 官方文档、官方 SDK、原始 API 规范、官方 Changelog、官方 RFC |
| **L2** | 独立可验证来源 | 独立第三方技术报告、学术论文、权威机构研究（RAND、NIST 等）|
| **L3** | 社区主流实现 | 高依赖数量的开源项目、活跃维护的社区实现 |
| **L4** | 二手经验来源 | 工程师博客、技术文章、论坛讨论、X/Twitter 观点 |

**混用规则**：
- 核心结论必须有 L1 或 L2 来源支撑
- L3/L4 只能作为补充和佐证
- 单独的 L4 来源不能构成结论，只能标为「待验证假设」

---

## 按问题类型选渠道

### 平台 / API 接入类

**触发词**：「怎么接入 X」「有没有 X 的 API」「X 平台怎么用」

**首选渠道（优先级顺序）**：
1. 官方文档 / 开发者门户（L1）
2. 官方 SDK / 官方示例代码（L1）
3. 官方 Changelog / 发布说明（L1）
4. 官方 GitHub 仓库的 issues 和 discussions（L1/L3）

**特殊规则 — 合规前置拦截**：
- 第一个判断：这个接入方式是官方通道还是逆向协议？
- 逆向协议 / 非官方协议 → 直接排除，不参与任何比较
- 这不是评分项，是拦截器——合规性通不过，其他维度无意义

**失败模式**：搜到第一个「能用」的实现方案就开始深入，没有先判断是否是官方通道。

---

### 技术选型类

**触发词**：「用哪个库」「A 和 B 怎么选」「推荐什么框架」

**首选渠道**：
1. 官方文档（了解各方案的设计意图和边界）（L1）
2. GitHub（评估维护状态）（L3）
3. 独立技术报告 / 基准测试（L2）

**GitHub 质量评估指标**（不用 stars）：
- `dependent repositories 数量`：有多少项目依赖它（比 stars 更能反映实际使用）
- `issue 响应速度`：最近 issues 是否有维护者回复
- `最近 commit 频率`：过去 3 个月有没有活跃维护
- `bus factor`：核心贡献者是否过度集中（单人维护风险）

**禁止**：用 stars 数量作为选型的主要依据。高 stars 可能只反映知名度，不反映质量。

---

### 可行性验证类

**触发词**：「X 能不能做到」「技术上可行吗」「有没有技术限制」

**首选渠道**：
1. 官方文档（确认功能边界）（L1）
2. 官方示例 / 官方 Cookbook（L1）
3. 最小可运行示例（社区验证）（L3）

**Spike 思路**：找到最小可验证路径——不是要找「完整解决方案」，而是找「能证明技术边界存在或不存在」的最小证据。

**注意**：可行性的最终验证通常需要 PoC，二手调研只能给「理论上可行」的置信度。遇到暗知识缺口（R10）时，应建议直接做 PoC。

---

### 竞品 / 参考实现类

**触发词**：「别人怎么做的」「这个产品怎么实现的」「有没有类似项目」

**首选渠道**：
1. 公开代码库（GitHub）（L3）
2. 官方博客 / 工程师博客（L2/L4）
3. 产品文档 / changelog（L1）
4. Product Hunt、App Store 用户评论（L4，弱证据）

**关键规则**：先向用户确认参考对象名，再开始搜索。

> 「你说的参考项目是 [X]，我确认一下——是 [具体仓库/产品名] 吗？」

这条规则来自真实失败案例：要看 clawwork 项目，结果找到了 styleclaw。

---

### 学术方法 / 算法 / 评测类

**触发词**：「这个算法原理是什么」「有没有 benchmark」「学界怎么看」

**首选渠道**：
1. arXiv、Semantic Scholar、Google Scholar（L2）
2. Papers with Code（L2/L3）
3. 官方技术报告（L1/L2）
4. OpenReview（L2）

**注意**：论文发表日期重要——AI/ML 领域 3 年以上的论文结论可能已经被推翻。

---

### 趋势 / 社区观点类

**触发词**：「最近大家怎么用」「业界有什么新实践」「这个方向热不热」

**首选渠道**：
1. Hacker News（L4）
2. Reddit 相关 subreddit（L4）
3. X / Twitter 技术讨论（L4）
4. 技术 Newsletter（L4）

**强制规则**：
- 所有趋势类来源默认标注为「弱证据」
- 单独不构成结论，必须有 L1/L2 交叉验证才能升级
- 「大家都在用 X」不是「X 适合你的场景」的理由

---

## 来源质量快速检查

每条信息进入假设矩阵之前，快速回答：

1. **发布者是谁？** 官方组织 / 独立研究者 / 从业者 / 匿名？
2. **发布时间？** 技术类信息是否过期（API 接入类 > 1 年需要验证时效性）？
3. **利益关系？** 发布者是否有明显利益倾向（厂商对比自家产品）？
4. **可验证性？** 结论是否可以通过其他渠道独立验证？

通不过快速检查的来源降级到 L4，标注为「待验证」。
