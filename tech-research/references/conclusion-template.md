# 有立场结论模板

## 核心结构

来自情报分析领域 SAT（Structured Analytic Techniques）的 Key Judgments 规范。

```
推荐：[X]（置信度：高 / 中 / 低）
  核心依据：[2-3 句，说明为什么选 X]，来源 [L1/L2]

排除：
  - [Y]：[具体排除理由，不是「Y 不如 X」而是「Y 在 [具体条件] 下不可行，因为 [证据]」]
  - [Z]：[同上]

前提：以上结论建立在 [假设 A] 成立的基础上

颠覆条件：如果出现 [具体信号 B]，这个结论需要重新评估

待验证：[单一来源或只有 PoC 才能确认的结论，标注为「待验证假设」]
```

---

## 置信度标签定义

| 置信度 | 条件 |
|--------|------|
| **高** | ≥2 个独立 L1/L2 来源支持，无重大反对证据，核心假设已验证 |
| **中** | 有 L1/L2 来源，但存在未解决的竞争假设，或部分依赖 L3 |
| **低** | 主要来自 L3/L4，或核心假设未经验证，或只有单一来源 |

置信度「低」不代表结论错，代表「需要额外验证才能提高信心」。

---

## 颠覆条件（必填）

**为什么是必填项**：没有颠覆条件的结论等于把适用边界隐藏了——形式上有立场，实际上还是把「这个结论适不适合你」的判断推给了用户。

好的颠覆条件：具体、可观察、和结论有因果关系
- ✓「如果团队在 3 个月内扩大到 10 人以上，方案 B 的运维成本会超过方案 A 的性能成本」
- ✓「如果 Tencent iLink 在 2026 年底关闭公测，这个方向需要重新评估」
- ✗「如果情况发生变化，结论可能需要调整」（无意义，没有具体信号）

## 好的结论 vs 差的结论

### 差的结论（方案矩阵 + 无立场）

```
方案对比：

| 方案 | 优点 | 缺点 |
|------|------|------|
| A    | 性能好 | 成本高 |
| B    | 成本低 | 性能差 |
| C    | 兼容性强 | 维护复杂 |

建议结合业务需求和团队情况综合考虑。
```

问题：把判断成本全部转嫁给用户，调研等于没做。

---

### 好的结论（有立场 + 颠覆条件）

```
推荐：Tencent iLink AI Bot API（置信度：高）

核心依据：
这是腾讯官方提供的微信 Bot 接入通道，无需企业微信资质，
个人开发者可申请，合规风险为零。官方 SDK 有 Go/Python 实现，
接入文档完整。来源：官方文档 L1，clawwork 参考实现 L3。

排除：
- gewechat / WeChatPadPro：逆向协议，违反微信 ToS，存在封号风险——
  合规前置拦截，不参与进一步比较
- 企业微信服务号：需要企业资质，个人开发者无法使用

前提：团队能完成 iLink Bot 的申请流程（当前处于公测阶段）

颠覆条件：如果 iLink 公测结束后不对外开放，或申请需要企业资质，
需要切换到企业微信方案

待验证：
- iLink Bot 的消息频率限制（官方文档未明确，需要实际测试）
- 公测申请的审核周期（无法从二手资料确认）
```

## 矩阵的合理使用边界

矩阵不是永远错的，有一个合理使用场景：

**合理**：用户明确说「帮我比较多个方向，决策权在我，因为团队有不同的技术背景偏好」

**即便如此，矩阵后必须附立场**：
```
[矩阵表格]

如果是我：在你描述的约束（3 人团队、6 个月、无专职运维）下，我选 B，
原因是运营成本可控性比性能指标更重要——如果你的约束不同，告诉我哪里不一样。
```

**不合理**：AI 不知道该推荐什么，用矩阵回避判断

## 暗知识缺口下的结论处理

当终止条件 R10 触发（不确定性核心是情境变量）时，结论格式变为：

```
当前可以确认：[基于公开信息能做出的判断]（置信度：中）

无法通过二手调研确认：[情境变量 X]

建议下一步：[具体 PoC 或追问建议]
- 做什么：[最小验证动作]
- 验证什么：[要回答的具体问题]
- 预计成本：[时间/资源估算]
```

这比「调研结果不够完整，需要更多信息」更有价值——它明确告诉用户调研到了哪里，下一步该做什么。
