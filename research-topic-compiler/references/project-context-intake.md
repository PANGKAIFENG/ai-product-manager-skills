# Project Context Intake

## 目的

产品决策型研究必须绑定本地项目上下文。如果只做外部竞品报告，缺少"这个项目现在到底在哪里、该怎么改"的判断。

## 触发时机

Product Decision Research Mode 启动时自动执行。

## 读取优先级

按以下顺序查找项目上下文：

1. 用户当前消息中提供的项目信息
2. 当前工作目录下的 `PROJECT_CONTEXT.md` 或 `CLAUDE.md`
3. 项目下的 `research/` 或 `discussions/` 或 `decisions/` 目录
4. 相关代码入口（如 `skillMarketCatalog.ts`、`starterCatalog.ts`）
5. 用户明确指定的文件路径
6. 如果以上都不可用，向用户追问

## 需要获取的上下文

| 字段 | 含义 | 必要性 |
|------|------|--------|
| 产品当前定位 | 一句话描述当前产品是什么 | 必须 |
| 目标用户 | 谁在用、按什么维度分群 | 必须 |
| 当前能力 | 已有什么功能/模块/技能 | 必须 |
| 已知限制 | 技术债、架构约束、合规边界 | 重要 |
| 已有决策 | 之前已经确定的产品方向 | 重要 |
| 已排除方向 | 之前明确否定的方向 | 重要 |
| 本轮决策 | 这次研究要服务什么决策 | 必须 |
| 产品层级 | SaaS / Desktop / Mobile / IM / 混合 | 重要 |
| 商业模型 | 免费/付费/分层/企业 | 可选 |

## 输出分层

研究输出中区分信息来源性质：

| 标签 | 含义 |
|------|------|
| `Verified local context` | 从项目文件中直接确认的事实 |
| `External evidence` | 外部公开资料支持的结论 |
| `Current judgment` | 项目团队当前持有的判断（可能变） |
| `Inference` | 基于证据的推断 |
| `Open decision` | 仍未决定的问题 |

## 上下文过期处理

如果读取到的项目文件日期较早（>30 天），标注：

> 「本地上下文来自 [日期]，可能已过期。以下判断基于当时的项目状态。」

## 不追问原则

能从本地文件发现的信息不问用户。只在以下情况追问：

- 答案会改变研究范围（目标用户不同 → 研究竞品不同）
- 答案涉及访问权限（某个系统需要登录）
- 答案无法从代码/文档推断（隐性战略判断）
