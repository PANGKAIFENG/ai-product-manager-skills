# Post-Research Exits

## 目的

产品决策型研究不是终点。研究结束后应能自然转成下游产物，推动产品推进。

## 可用出口

| 出口 | 适合什么时候 | 输出格式 |
|------|------------|---------|
| Top Starter Scenes | 从候选池挑出首批展示的场景 | 排序列表 + 每个场景的 1-sentence 描述 |
| PRD 输入 | 候选进入正式产品规划 | PRD 片段：用户故事、验收标准、输入输出、权限 |
| Demo Beachhead | 选 3-5 个最适合演示的候选 | 演示脚本：场景、输入、预期输出、wow moment |
| Eval Checklist | 定义如何评估候选的质量 | 评估维度、评分标准、通过门槛 |
| Skill Registry Schema | 为候选定义上架数据结构 | 字段定义、必填/选填、展示规则 |
| 上新审核流程 | 定义候选如何从 backlog 到上架 | 审核步骤、角色、门禁、回滚条件 |
| 路线图 | 按时间或优先级排列候选 | Phase 1/2/3 分组 + 每 phase 交付定义 |

## Top Starter Scenes 格式

```markdown
**Top [N] Starter Scenes**

| Rank | Name | Customer | Persona | Scenario 1-liner | Why starter | Surface |
|------|------|----------|---------|------------------|-------------|---------|
| 1 | [名称] | [客户类型] | [角色] | [一句话描述] | [为什么适合首批展示] | [SaaS/Desktop] |
```

选择标准：
- 易理解（30 秒内看懂）
- 可演示（有清晰的输入→输出）
- 有 wow moment（结果让人惊喜）
- 差异化强（不是通用 AI 都能做的）

## PRD 输入格式

```markdown
**PRD Input: [候选名称]**

用户故事：作为 [角色]，我想要 [做什么]，以便 [获得什么价值]

输入：
- [输入 1]：[格式/来源]
- [输入 2]：[格式/来源]

输出：
- [交付物]：[格式/规格]

验收标准：
- [ ] [标准 1]
- [ ] [标准 2]

权限边界：
- [需要什么权限]
- [人审确认点]

底层依赖：
- [capability 1]
- [capability 2]

风险：
- [风险 1]：[缓解方式]
```

## Demo Beachhead 格式

```markdown
**Demo Beachhead: [候选名称]**

场景设定：[用户是谁、在做什么]
演示输入：[具体输入物]
预期输出：[演示中看到什么]
Wow moment：[哪个瞬间让人印象深刻]
演示时长：[分钟]
需要准备：[数据/素材/环境]
```

## 触发规则

- 用户明确要求"给我 starter""输出 PRD""demo 脚本"→ 直接输出对应格式
- 产品决策型研究完成时 → 默认建议可用出口，但不自动全部输出
- 候选池 > 10 个时 → 建议先做 Top Starter 筛选，再做 PRD/Demo

## 不自动触发的出口

以下出口需要用户明确要求：

- 上新审核流程（涉及团队角色和组织决策）
- 路线图（涉及时间承诺和资源分配）
- Eval Checklist（需要和团队对齐评估标准）
